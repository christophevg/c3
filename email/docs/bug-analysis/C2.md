# Bug Analysis: C2 — RuntimeError Misclassified as Rate Limits

**Bug ID:** C2
**Status:** Fixed ✅
**Created:** 2026-04-28
**Fixed:** 2026-04-28
**Affected Files:** `server.py`, `connections/pool.py`

---

## Summary

All `RuntimeError` exceptions raised anywhere in the email MCP stack are caught by tool handlers in `server.py` and mapped to the user-facing message **"Rate limit exceeded. Please try again later."** This includes authentication failures, IMAP protocol errors, missing messages, folder selection failures, and SMTP send failures — none of which are rate-limit errors.

## Expected vs Actual Behavior

| Scenario | Expected Error | Actual Error |
|----------|---------------|--------------|
| IMAP auth fails | "Authentication failed" or generic error | "Rate limit exceeded" |
| Message ID not found | "Failed to fetch message" or "not found" | "Rate limit exceeded" |
| Folder selection fails | "Failed to select folder" | "Rate limit exceeded" |
| SMTP send fails | "Failed to send email" | "Rate limit exceeded" |
| Rate limit actually hit | "Rate limit exceeded" | "Rate limit exceeded" |

## Root Cause Analysis

### Source: `connections/pool.py`

```python
if not await imap_limiter.acquire(account_name):
    raise RuntimeError("IMAP rate limit exceeded for account")  # line 60
```

The connection pool uses the generic `RuntimeError` to signal rate-limit exhaustion.

### Sink: `server.py` tool handlers

Every tool handler contains this pattern:

```python
except RuntimeError:
    raise ToolError("Rate limit exceeded. Please try again later.")
```

This catches `RuntimeError` from ANY source — not just the pool.

### Other RuntimeError sources

`imap/client.py` raises `RuntimeError` for:
- Authentication failure (line 72)
- Folder LIST failure (line 92)
- Folder SELECT failures (lines 118, 143, 184, 227, 254, 277, 316)
- Search failure (line 153)
- Fetch failure (lines 191, 322)
- Copy failure (line 233)

`smtp/client.py` raises `RuntimeError` for:
- Send failure (line 197)

## Proposed Fix

1. **Create custom `RateLimitError`** — a dedicated exception class for rate-limit violations.
2. **Raise `RateLimitError` from pool** — replace `RuntimeError` in `connections/pool.py`.
3. **Catch `RateLimitError` specifically in server** — replace `except RuntimeError` with `except RateLimitError` in all tool handlers.
4. **Let other RuntimeErrors propagate** — they fall through to the generic `except Exception` handler, which returns an appropriate generic error.

## Test Strategy

- Unit test that pool raises `RateLimitError` when limiter rejects
- Unit test that server tools return "Rate limit exceeded" only for `RateLimitError`
- Unit test that other RuntimeErrors (e.g., auth failure) return generic error messages

## Risk Assessment

| Risk | Level | Mitigation |
|------|-------|------------|
| Missing a tool handler | Low | Search/replace all `except RuntimeError` in `server.py` |
| Custom exception import path | Low | Define in `safety/` module alongside rate limiter |
| Regression in actual rate limiting | Low | Test that limiter still triggers the correct message |

## Fix Summary

1. Created `RateLimitError(RuntimeError)` custom exception in `connections/pool.py`
2. Changed `pool.py` to raise `RateLimitError` instead of `RuntimeError` when rate limits are exceeded
3. Replaced all 9 occurrences of `except RuntimeError` with `except RateLimitError` in `server.py` tool handlers
4. Other `RuntimeError`s from IMAP/SMTP clients now fall through to the generic `except Exception` handler, returning appropriate generic error messages
