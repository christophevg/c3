# Bug Analysis: C5 - SMTP Exception Chain Swallowing

## Summary

The SMTP client catches `aiosmtplib.SMTPException` but discards the original exception, losing all error details that would aid debugging. The fix requires preserving the exception chain using `raise ... from e` syntax.

## Location

- **File**: `src/email_mcp/smtp/client.py`
- **Lines**: 204-205

## Expected vs Actual Behavior

### Expected Behavior
When an SMTP error occurs, the `RuntimeError` should preserve the original exception chain so developers and operators can:
- See the underlying SMTP error type (authentication failure, connection refused, etc.)
- Access the full traceback for root cause analysis
- Debug issues without needing server-side logs

### Actual Behavior
The original exception `e` is caught but never used:

```python
except aiosmtplib.SMTPException as e:
  raise RuntimeError("Failed to send email. Check server logs for details.")
```

This:
- Loses the `SMTPException` type and its specific attributes
- Discards all error context (server response, error codes)
- Forces operators to check external logs instead of the traceback

## Root Cause

Python's exception handling by default creates a new exception that replaces the original. To preserve the chain, the `from` keyword must be used explicitly. This is a common oversight when converting library exceptions to domain-specific exceptions.

## Proposed Fix Approach

Change line 205 to use exception chaining:

```python
except aiosmtplib.SMTPException as e:
  raise RuntimeError("Failed to send email. Check server logs for details.") from e
```

This minimal change:
- Preserves `e.__cause__` reference to the original exception
- Shows both exceptions in the traceback
- Allows exception handlers up the chain to inspect the original error
- Maintains the generic user-facing message while preserving debugging context

## Test Strategy

### Test Cases

1. **Exception chain is preserved**
   - Mock `aiosmtplib.send` to raise `aiosmtplib.SMTPException("Server error")`
   - Call `client.send_email()` and expect `RuntimeError`
   - Verify `exc_info.value.__cause__` is the original `SMTPException`
   - Verify `exc_info.value.__cause__.args[0]` contains the original message

2. **Traceback shows both exceptions**
   - Verify the string representation includes context from both exceptions
   - The RuntimeError message should be the primary message
   - The SMTPException should appear in `__cause__`

### Test Implementation Pattern

```python
class TestSMTPExceptionChaining:
  """Tests that SMTP exceptions preserve exception chain (C5 bug fix)."""

  async def test_smtp_exception_chain_preserved(self, mock_account):
    """SMTPException should be chained to RuntimeError via 'from'."""
    client = SMTPClient(mock_account)

    with patch("email_mcp.smtp.client.aiosmtplib.send") as mock_send:
      mock_send.side_effect = aiosmtplib.SMTPException("Authentication failed")

      with pytest.raises(RuntimeError) as exc_info:
        await client.send_email(
          to=["user@test.com"],
          subject="Test",
          body="Body",
        )

      # Verify exception chain is preserved
      assert exc_info.value.__cause__ is not None
      assert isinstance(exc_info.value.__cause__, aiosmtplib.SMTPException)
      assert "Authentication failed" in str(exc_info.value.__cause__)

  async def test_smtp_exception_message_unchanged(self, mock_account):
    """RuntimeError message should remain generic for user-facing output."""
    client = SMTPClient(mock_account)

    with patch("email_mcp.smtp.client.aiosmtplib.send") as mock_send:
      mock_send.side_effect = aiosmtplib.SMTPException("Any error")

      with pytest.raises(RuntimeError) as exc_info:
        await client.send_email(
          to=["user@test.com"],
          subject="Test",
          body="Body",
        )

      assert "Failed to send email" in str(exc_info.value)
```

## Risk Assessment

### Is This a Breaking Change?

**No.** This is not a breaking change:

1. **Exception Type Unchanged**: The raised exception remains `RuntimeError`
2. **Exception Message Unchanged**: The user-facing message is identical
3. **Public API Unchanged**: Method signatures and return types are unaffected
4. **Backward Compatible**: Code catching `RuntimeError` continues to work identically

### What Changes

- **Enhanced Debugging**: Tracebacks now show the full exception chain
- **New Attribute**: `RuntimeError.__cause__` is now populated (was `None`)
- **Traceback Format**: Exception output is longer (shows chained exception)

### Low Risk

This is a safe, minimal fix that only adds information. No behavior is removed or changed for callers who only catch the `RuntimeError`.
