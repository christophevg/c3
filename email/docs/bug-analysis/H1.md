# Bug Analysis: H1 - Non-atomic move_message Operation

**Bug ID**: H1
**Location**: `src/email_mcp/imap/client.py:221-246`
**Priority**: P2 (High)
**Date**: 2026-04-28
**Analyst**: Functional Analyst

---

## Summary

The `move_message()` method implements email move operations using a sequence of COPY → STORE+Deleted → EXPUNGE commands. This sequence is not atomic and can leave the system in an inconsistent state where the message exists in both source and destination folders if an intermediate step fails.

---

## Expected vs Actual Behavior

### Expected Behavior

A move operation should be atomic: either the message is fully moved to the destination folder, or it remains in the source folder. No intermediate state where the message exists in both locations should be possible.

### Actual Behavior

The current implementation uses a non-atomic sequence:

```python
# Step 1: Copy to destination
status, _ = await client.copy(message_id, dest_folder)
if status != "OK":
  raise RuntimeError(f"Failed to copy message: {status}")

# Step 2: Mark as deleted in source
await client.store(message_id, "+FLAGS", "\\Deleted")

# Step 3: Permanently remove from source
await client.expunge()
```

**Failure scenarios:**

| Failure Point | Result | Data Integrity |
|---------------|--------|----------------|
| COPY fails | Exception raised, no changes | OK - message still only in source |
| STORE fails after COPY succeeds | Exception raised, message in BOTH folders | **VIOLATION** |
| EXPUNGE fails after COPY succeeds | Exception raised, message in BOTH folders | **VIOLATION** |
| Network disconnect after COPY, before STORE | Message in BOTH folders | **VIOLATION** |

---

## Root Cause

### Primary Cause

The IMAP protocol does not have a built-in atomic move command in the base specification (RFC 3501). The traditional approach requires multiple commands, which creates a window for partial completion.

### Secondary Causes

1. **No capability detection**: The code does not check if the server supports the MOVE extension (RFC 6851)
2. **No fallback documentation**: The limitation of the fallback approach is not documented
3. **No transaction boundary**: There is no rollback mechanism if intermediate steps fail

### Historical Context

RFC 6851 (January 2013) introduced the IMAP MOVE extension, providing an atomic MOVE command. Most modern IMAP servers support this extension:
- Gmail: Supported
- Outlook/Office365: Supported
- iCloud: Supported (as of 2024)
- Fastmail: Supported
- Dovecot: Supported (v2.2+)
- Courier: Not supported

## Proposed Fix Approach

### Strategy: Capability Detection with Graceful Fallback

The fix involves checking server capabilities during connection and using the atomic MOVE command when available.

### Implementation Pattern

```python
class IMAPClient:
  def __init__(self, account: EmailAccount) -> None:
    self.account = account
    self._client: IMAP4_SSL | None = None
    self._selected_folder: str | None = None
    self._connect_lock = asyncio.Lock()
    self._operation_lock = asyncio.Lock()
    self._capabilities: set[str] = set()  # NEW: cached capabilities

  async def connect(self) -> IMAP4_SSL:
    async with self._connect_lock:
      if self._client is not None:
        return self._client

      # ... existing connection code ...

      # Query and cache capabilities
      response = await self._client.capability()
      if response.result == "OK":
        self._capabilities = set(
          line.decode().upper().split()
          for line in response.lines
        )

      return self._client

  def has_capability(self, name: str) -> bool:
    """Check if server supports a capability."""
    return name.upper() in self._capabilities

  async def move_message(
    self,
    message_id: str,
    source_folder: str,
    dest_folder: str,
  ) -> bool:
    """Move a message between folders atomically if server supports MOVE.

    Uses RFC 6851 MOVE extension when available for atomic operation.
    Falls back to COPY+STORE+EXPUNGE with documented non-atomic limitation.
    """
    async with self._operation_lock:
      client = await self.connect()
      status, data = await client.select(source_folder)
      if status != "OK":
        raise RuntimeError(f"Failed to select folder {source_folder}: {status}")
      self._selected_folder = source_folder

      # Use atomic MOVE if server supports RFC 6851
      if self.has_capability("MOVE"):
        status, _ = await client.move(message_id, dest_folder)
        if status != "OK":
          raise RuntimeError(f"Failed to move message: {status}")
        return True

      # Fallback: non-atomic COPY+STORE+EXPUNGE
      # WARNING: Not atomic - message may exist in both folders on failure
      status, _ = await client.copy(message_id, dest_folder)
      if status != "OK":
        raise RuntimeError(f"Failed to copy message: {status}")

      await client.store(message_id, "+FLAGS", "\\Deleted")
      await client.expunge()

      return True
```

### Capability Check Pattern

```python
# During connection, after authentication
response = await client.capability()
# response format: * CAPABILITY IMAP4rev1 MOVE UIDPLUS ...
# Parse and store for later use
```

### aioimaplib MOVE Support

aioimaplib provides the `move()` method for IMAP MOVE command. The method signature follows the same pattern as `copy()`:

```python
# aioimaplib move signature
await client.move(sequence_set, mailbox)
# Example: move message 1 to Archive folder
await client.move("1", "Archive")
```

**Note**: The MOVE command in aioimaplib was added in version 1.0.0 (current dependency is `>=1.0.0`).

## Risk Assessment

### Severity: High

**Data Integrity Risk**: Messages can be duplicated across folders, violating the expected single-instance semantic of email. This can lead to:
- Users seeing duplicate emails
- Incorrect email counts
- Confusion about which folder contains the "real" email
- Potential data loss if user assumes move succeeded and deletes from source

### Likelihood: Medium

**Failure Probability**: The risk manifests when:
1. Network connection drops between COPY and STORE/EXPUNGE
2. Server crashes or times out during the sequence
3. Client process is killed between commands
4. Server resource limits hit during EXPUNGE

In typical usage with stable networks, the window is small (tens to hundreds of milliseconds), but in edge cases (mobile networks, server load spikes), the probability increases.

### Impact Analysis

| Impact Area | Severity | Notes |
|-------------|----------|-------|
| Data Integrity | High | Duplicated emails |
| User Experience | Medium | Confusion from duplicates |
| Audit Trail | Medium | Unclear which folder holds original |
| Compliance | Low-Medium | Some regulations require auditability of email moves |

### Mitigation Priority

**Immediate (P2)**: Implement capability detection and use MOVE when available.

**Long-term**: Document the fallback limitation and consider client-side cleanup for detected duplicates (outside scope of this fix).
