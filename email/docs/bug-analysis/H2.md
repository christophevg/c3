# Bug Analysis: H2 - Broad Exception Handling During Authentication

## Summary

**Bug ID**: H2
**Severity**: High (P2)
**Location**: `src/email_mcp/imap/client.py:76-78`
**Impact**: Misleading error messages, degraded debugging experience, potential security audit noise

## Bug Validation

### Current Code (Confirmed)

```python
# Lines 62-78 in client.py
try:
  if self.account.auth_method == "oauth2" and self.account.oauth2_token:
    token = self.account.oauth2_token.get_secret_value()
    auth_string = f"user={self.account.username}\x01auth=Bearer {token}\x01\x01"
    await self._client.authenticate("XOAUTH2", auth_string)
    log_auth_attempt(self.account.name, True, "oauth2")
  elif self.account.password:
    await self._client.login(
      self.account.username,
      self.account.password.get_secret_value(),
    )
    log_auth_attempt(self.account.name, True, "password")
  else:
    raise ValueError("No credentials available")
except Exception as e:
  log_auth_attempt(self.account.name, False, self.account.auth_method, str(e))
  raise RuntimeError("Authentication failed. Check server logs for details.")
```

### Bug Confirmation: VALID

The bug is confirmed. The `except Exception as e:` clause catches ALL exceptions including network-level errors, timeouts, and protocol issues - all of which are logged as authentication failures and presented to users as "Authentication failed".

## Expected vs Actual Behavior

### Scenario Analysis

| Error Type | Expected Message | Actual Message | Audit Log |
|------------|------------------|----------------|-----------|
| Invalid credentials | "Authentication failed" | "Authentication failed" | AUTH_FAILURE (correct) |
| Network disconnected | "Connection error" | "Authentication failed" | AUTH_FAILURE (incorrect) |
| DNS resolution failed | "Connection error" | "Authentication failed" | AUTH_FAILURE (incorrect) |
| Connection timeout | "Connection timeout" | "Authentication failed" | AUTH_FAILURE (incorrect) |
| Protocol error | "Protocol error" | "Authentication failed" | AUTH_FAILURE (incorrect) |
| TLS handshake failed | "Connection error" | "Authentication failed" | AUTH_FAILURE (incorrect) |

### User Impact

1. **Misleading diagnostics**: User sees "Authentication failed" when the actual problem is network connectivity
2. **Wasted troubleshooting time**: Users may repeatedly check credentials when the issue is firewall/DNS
3. **Security audit noise**: Connection failures are logged as AUTH_FAILURE, potentially triggering security alerts
4. **No actionable feedback**: "Check server logs" is not actionable for end users

## Root Cause

### Exception Hierarchy Analysis

The code catches `Exception`, which is the base class for all non-system-exiting exceptions. This includes:

1. **aioimaplib.Abort** - Raised for:
   - Connection abort
   - Protocol violations
   - Unexpected server behavior
   - TLS handshake failures

2. **aioimaplib.Error** - Base class for aioimaplib errors:
   - Command syntax errors
   - Response parsing errors

3. **asyncio.TimeoutError** - Raised for:
   - Connection timeout during `wait_hello_from_server()`
   - Command timeout

4. **TimeoutError** (built-in) - May be raised by socket operations

5. **ConnectionError** (built-in) - Network disconnection

6. **OSError** subclasses:
   - `socket.gaierror` - DNS resolution failure
   - `ConnectionRefusedError` - Server not listening
   - `ConnectionResetError` - Connection dropped

7. **ssl.SSLError** - TLS certificate/validation errors

8. **ValueError** - Missing credentials (intentionally raised)

### The Problem

All of these exceptions are caught by the single `except Exception` handler and:
1. Logged as authentication failures via `log_auth_attempt(..., False, ...)`
2. Re-raised as generic `RuntimeError("Authentication failed...")`

This violates the principle of specific exception handling and provides no distinction between:
- "You typed the wrong password" (user error)
- "The server is unreachable" (infrastructure error)
- "The connection timed out" (network error)

## Proposed Fix Approach

### Exception Handling Strategy

Split the catch block into specific handlers with appropriate error messages:

```python
try:
  # ... authentication code ...
except ValueError as e:
  # Missing credentials - configuration error
  log_auth_attempt(self.account.name, False, self.account.auth_method, str(e))
  raise RuntimeError(f"Configuration error: {e}")
except asyncio.TimeoutError:
  # Connection timeout during authentication
  raise RuntimeError("Connection timed out while authenticating. Check server availability.")
except TimeoutError:
  # Generic timeout (from socket layer)
  raise RuntimeError("Connection timed out. Check network connectivity.")
except ConnectionError as e:
  # Network-level disconnection
  raise RuntimeError(f"Connection lost: {e}")
except OSError as e:
  # DNS failures, connection refused, etc.
  if isinstance(e, socket.gaierror):
    raise RuntimeError(f"DNS resolution failed for {self.account.imap_host}")
  raise RuntimeError(f"Network error: {e}")
except ssl.SSLError as e:
  # TLS certificate/validation errors
  raise RuntimeError(f"TLS error: {e}. Check server certificate.")
except aioimaplib.Abort as e:
  # Protocol-level abort (server disconnected, protocol error)
  raise RuntimeError(f"Protocol error: {e}")
except aioimaplib.Error as e:
  # IMAP-specific errors (bad command, response error)
  log_auth_attempt(self.account.name, False, self.account.auth_method, str(e))
  raise RuntimeError("Authentication failed. Check server logs for details.")
except Exception as e:
  # Unexpected errors - log full traceback for debugging
  logger.error(f"Unexpected error during auth: {e}\n{traceback.format_exc()}")
  raise RuntimeError("An unexpected error occurred. Check server logs for details.")
```

### Exception Priority Order

Exceptions should be caught in this order (most specific first):

1. `ValueError` - Configuration issues (our own exception)
2. `asyncio.TimeoutError` - Async timeout
3. `TimeoutError` - Generic timeout
4. `ConnectionError` - Network disconnection
5. `ssl.SSLError` - TLS issues
6. `aioimaplib.Abort` - Protocol abort
7. `aioimaplib.Error` - IMAP errors (includes auth failures)
8. `OSError` - Network/OS errors (must be last in network hierarchy)
9. `Exception` - Fallback for unexpected errors

### Key Decisions

1. **Only log AUTH_FAILURE for actual auth errors**: `aioimaplib.Error` subclasses that represent authentication failures
2. **Use specific error messages**: Each exception type gets a helpful, actionable message
3. **Preserve exception chain**: Use `raise ... from e` for debugging
4. **Add imports**: Will need `import ssl`, `import socket`, `import traceback`, `import asyncio` (if not already present)
